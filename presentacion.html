Ponente 3: Arquitectura Técnica

El Motor: Linux y Docker

El Servidor (Linux)

Es el "edificio blindado" de la farmacia. Se elige Linux (como Ubuntu Server) por su robustez, seguridad demostrada en servidores web y gestión eficaz de recursos (RAM/CPU) en entornos críticos sin parches sorpresa que reinicien el sistema a las 11:00 AM.

Contenerización (Docker)

Son los "armarios estancos". Aislamos el software de recetas, del software de stock, del software de RRHH de la farmacia. Si el programa de RRHH falla o sufre un bug catastrófico, el contenedor de las recetas electrónicas ni se inmuta al estar encapsulado.

Ponente 3: Arquitectura Técnica

Gestión de Tráfico y Seguridad

Puntos Clave (Guía Ponente 3):
  • Nginx (El recepcionista): Recibe las miles de conexiones desde las distintas cajas registradoras y las envía ordenadamente a los contenedores Docker para evitar que el servidor se ahogue (balanceo).
  • Seguridad por Variables (.env): Las contraseñas para conectarse con el Ministerio NO ESTÁN en el código HTML o JS de los desarrolladores. Se inyectan externamente en el entorno del servidor para que nadie pueda robarlas.
  • Cifrado (HTTPS): Uso de certificados SSL (Let's Encrypt). Obligatorio para que si alguien hace sniffing en el Wi-fi local, no lea que DNI retiró antidepresivos.
Ponente 4: El Impacto Daw

¿Y si ocurre un Error 500?

Internal Server Error

Un "Código Bugueado" en una web corporativa solo causa pérdida de clientes.
Un "Código Bugueado" en farmacia interrumpe la toma de medicación vital de miles de ciudadanos en la comunidad.

Ponente 4: El Impacto Daw

Medidas Críticas de DAW

Disponibilidad 99.99% (Alta Disponibilidad)

Aprovechamos Docker Compose y orquestadores para levantar inmediatamente un contenedor idéntico si el original muere por memoria saturada.

Estrategia de Backups y CRON

Tareas automatizadas en el servidor (Cronjobs) que diariamente hacen un volcado seguro de la base de datos a un disco externo físico por si hubiese un ataque Ransomware.

Ponente 4: El Impacto Daw

La Barrera: Entornos (Prod vs Dev)

Explicar al aula:
  • ¿Por qué caen páginas vitales estatales? A menudo porque un desarrollador subió código "para probar".
  • Entorno Desarrollo: El PC del programador (con bases de datos inventadas, "John Doe").
  • Entorno Staging (Pruebas): Un servidor idéntico al real donde revisa el titular de farmacia, aislado.
  • Entorno Producción: El servidor final. NUNCA se toca código aquí a mano, solo se hace despliegue por pipelines (CI/CD) automáticos y seguros.
